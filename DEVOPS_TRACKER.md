## 1. Infra status

### 1.1 Outage 2026-07-07 → 2026-07-11 — RESOLVED

`alecrae.com`, `mail.alecrae.com`, and `status.alecrae.com` were found
returning **HTTP 503**. The box's own `fleet-check.sh` journal shows the
outage started by 2026-07-07 09:25 UTC. Root cause was three stacked
failures on the Jarvis box (66.42.121.161): (1) when Coolify/Traefik took
over ports 80/443, **no Traefik route for alecrae was ever created** — all
requests hit the default 503 catch-all, so the site was down even while
the services ran; (2) `alecrae-api`/`alecrae-web` were then **stopped and
disabled** on Jul 7; (3) both services **bound to 127.0.0.1** (set in
`/opt/alecrae/.env`, which overrides the systemd unit's `HOST=` because
Bun auto-loads `.env`), unreachable from the proxy container regardless.

**Fixed 2026-07-11:** services enabled + started; append-only
`/data/coolify/proxy/dynamic/alecrae.yaml` route added (web/mail/www →
`10.0.1.1:4200`, api → `10.0.1.1:4100`, same pattern as the working
gatetest services); bind changed to `0.0.0.0` in `.env` + unit files; two
ufw allows added for the proxy network only (4100/4200 stay publicly
firewalled). All four domains verified 200 externally.
`status.alecrae.com` remains 503 — no status app is deployed on 161 and no
route exists for it (known, low priority). **Postmortem still owed** to
`docs/postmortems/` per CLAUDE.md's Emergency Protocol; the durable lesson
is §1.7 (no staging, single box, and nothing alerting on the fleet-check
503s for three days).

### 1.7 Reliability gap: no staging environment

Tonight's outage exposed that there is **no staging environment anywhere in
this stack** — production is a single box with a manual `git pull` ritual,
so every merge to `main` is one deploy away from being live with nothing in
between to catch breakage. The box in question reportedly serves multiple
products beyond alecrae.com, which raises the stakes further. **Recommend a
dedicated, separate conversation to design a proper staging tier** (likely
spanning infrastructure beyond this repo) rather than architecting it
inline here — flagged, not actioned.

### 1.8 Deploy drift + missing Redis — RESOLVED 2026-07-14

Found during the 2026-07-14 morning audit: the box had served a **July-4
build for 10 days**. An unpushed on-box hotfix (`e480456`) made the checkout
diverge, so `scripts/box-deploy.sh`'s `git merge --ff-only` refused to pull
— correctly, but silently: nothing compared box HEAD to `origin/main`
(Known Issue #78). Separately, **redis-server had never been installed** on
Jarvis despite docs claiming otherwise — the API logged constant
`ECONNREFUSED 127.0.0.1:6379` and `/v1/health` reported redis down.

**Fixed same morning (Craig-authorized):** hotfix confirmed subsumed by
main → `git reset --hard origin/main`; `redis-server` installed and enabled
(binds 127.0.0.1); full deploy ritual run (install → migrate → web build →
restart → health checks). Lessons: never commit on the box; a failed
`--ff-only` is a stop-and-reconcile signal; drift detection still needed
(#78).
